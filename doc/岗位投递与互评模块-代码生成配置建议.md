# 岗位投递与互评模块-代码生成配置建议（详细版）

本文用于配合 `RuoYi` 代码生成器尽量“一次生成到位”，减少二次改造工作量。

适用边界：

- 本期仅覆盖最小闭环：`发布岗位 -> 投递 -> 录用 -> 企业标记完成 -> 双向互评`；
- 互评闸门：仅当 `t_job_application.application_status='2'`（企业已完成）后开放；
- 公告复用 `sys_notice`、`sys_notice_read`，不新建公告表。

## 1. 导入与生成顺序（建议固定）

1. 执行 `backend/sql/student_flow_phase1.sql`。
2. 若数据库里曾导入过旧业务菜单，先执行 `backend/sql/rbac/V20260415_00__legacy_menu_cleanup.sql`。
3. 再执行 `backend/sql/rbac/V20260414_01__role_split_menu_rbac.sql`（仅新建菜单/权限）。
3. 后台进入 `系统工具 -> 代码生成`，按依赖顺序导入：
   - `t_student_info`
   - `t_student_resume`
   - `t_enterprise_info`
   - `t_job_post`
   - `t_job_application`
   - `t_job_evaluation`
   - `t_job_collection`

> 说明：`t_job_collection` 放最后导入，避免你先生成“后台完整CRUD”，后续又要裁掉大量按钮和接口。

## 2. 代码生成通用配置模板

- 模块名：`biz`
- 业务包：`com.ruoyi.biz`
- 上级菜单：按角色分目录（企业端、管理端）；学生端通常为接口权限，不必强制生成后台菜单页
- 作者：你的姓名缩写
- 前端模板：`element-ui`
- 生成方式：先 `zip`，评审通过后再落地到项目

### 2.1 通用字段处理规则（减少后改）

- 以下字段默认不放“新增/编辑表单”：
  - 主键：`*_id`
  - 统计字段：`view_count`、`collect_count`、`apply_count`、`hire_count`、`complete_count`
  - 审计字段：`create_by/create_time/update_by/update_time`
  - 软删除字段：`del_flag`
- 以下字段默认“仅详情/列表展示，禁止前端直编”：
  - `application_status`、`process_time`、`complete_time`
  - `evaluation_time`
  - `enterprise_id`、`student_id`（由登录身份或关联记录推导）
- 查询默认值：
  - 时间字段统一用 `BETWEEN`（如 `deadline_time/apply_time/evaluation_time`）
  - 编码状态字段统一用 `EQ`
  - 名称描述字段统一用 `LIKE`

### 2.2 字典映射（务必在生成前确认存在）

- `t_student_info.gender` -> `sys_user_sex`
- `t_enterprise_info.audit_status` -> `enterprise_audit_status`
- `t_job_post.post_status` -> `job_post_status`
- `t_job_post.salary_unit` -> `job_salary_unit`（若未建字典，先按单选框固定值）
- `t_job_post.work_location_type` -> `job_location_type`
- `t_job_post.work_type` -> `job_work_type`
- `t_job_application.application_status` -> `job_application_status`
- `t_job_evaluation.evaluation_type` -> `job_evaluation_type`

## 3. 生成前数据库自检清单（建议执行）

在进入代码生成前，先完成以下检查：

1. 表是否完整存在：`t_student_info/t_student_resume/t_enterprise_info/t_job_post/t_job_application/t_job_collection/t_job_evaluation`。
2. 唯一约束是否生效：
   - `uk_application_post_student`（同学生同岗位仅一条投递）
   - `uk_collection_student_post`（同学生同岗位仅一条收藏）
   - `uk_eval_application_type`（同投递同评价类型仅一条互评）
3. 外键是否完整：投递、互评、收藏需可追溯到学生/企业/岗位/简历。
4. 字典是否已初始化：`job_post_status/job_application_status/job_evaluation_type/enterprise_audit_status`。
5. RBAC 权限点是否已初始化：`enterprise:*`、`student:*`、`admin:*`。
6. 审计字段与软删除字段是否齐全（确保后续分页、追溯、逻辑删除一致）。

建议自检 SQL（可按需执行）：

```sql
-- 表数量与命名检查
show tables like 't_%';

-- 关键唯一索引检查
show index from t_job_application;
show index from t_job_collection;
show index from t_job_evaluation;

-- 字典检查
select dict_type, count(1) as cnt
from sys_dict_data
where dict_type in ('job_post_status','job_application_status','job_evaluation_type','enterprise_audit_status')
group by dict_type;
```

## 4. 逐表详细生成配置

## 4.1 `t_job_post`（企业岗位管理）

- 查询条件：
  - `post_name`（LIKE）
  - `post_status`（EQ）
  - `deadline_time`（BETWEEN）
  - `create_time`（BETWEEN，可选）
- 列表展示：`post_name`、`salary_min/salary_max/salary_unit`、`work_address`、`post_status`、`deadline_time`、`apply_count`、`hire_count`、`complete_count`
- 新增/编辑表单必填：
  - `post_name`
  - `salary_min`、`salary_max`（校验 `salary_min <= salary_max`）
  - `salary_unit`
  - `work_location_type`、`work_address`
  - `work_type`、`work_time_desc`
  - `recruit_count`（>0）
  - `deadline_time`（应晚于当前时间）
  - `contact_name`、`contact_phone`
- 前端控件建议：
  - `salary_min/salary_max`：数字输入（保留2位小数）
  - `deadline_time`：日期时间控件
  - `requirement_desc`：多行文本
  - `contact_phone`：手机号正则校验
- 只读字段：`enterprise_id`、统计字段、审计字段、`del_flag`

## 4.2 `t_job_application`（投递处理）

- 查询条件：`post_id`（EQ）、`student_id`（EQ）、`application_status`（EQ）、`apply_time`（BETWEEN）
- 列表展示：`application_id`、`post_id`、`student_id`、`apply_time`、`application_status`、`process_time`、`complete_time`
- 新增表单仅保留：`post_id`、`resume_id`
  - `student_id` 由当前登录学生注入
  - `apply_time/application_status/process_time` 由服务端填充
- 编辑表单不建议开放通用“修改”
  - 企业端通过专用动作接口：录用、标记完成
- 只读字段：`application_status`、`process_time`、`complete_time`、审计字段、`del_flag`

## 4.3 `t_job_evaluation`（互评）

- 查询条件：`application_id`（EQ）、`evaluation_type`（EQ）、`evaluation_time`（BETWEEN）
- 列表展示：`application_id`、`evaluation_type`、`score`、`evaluation_time`、`content`
- 新增表单建议：
  - 可输入：`application_id`、`score`、`content`
  - 不建议前端直接选择：`evaluation_type/student_id/enterprise_id/post_id`（服务端推导）
- 校验：
  - `score` 在 `1~5`
  - 提交前必须校验投递状态为 `application_status='2'`
  - 同一 `application_id + evaluation_type` 不可重复
- 只读字段：`evaluation_time`、审计字段、`del_flag`

## 4.4 `t_job_collection`（收藏）

- 查询条件：`student_id`（EQ）、`post_id`（EQ）、`collection_time`（BETWEEN）
- 列表展示：`collection_id`、`student_id`、`post_id`、`collection_time`
- 不建议生成后台完整CRUD交互：
  - 后台保留查询即可
  - 新增/删除应由学生端接口驱动
- 只读字段：除查询外全部服务端维护

## 4.5 `t_student_resume`（简历）

- 查询条件：`student_id`（EQ）、`status`（EQ）、`update_time`（BETWEEN）
- 列表展示：`student_id`、`expected_position`、`expected_salary_min/max`、`status`、`update_time`
- 表单建议：
  - `education_experience/part_time_experience/self_intro` 使用多行文本
  - 薪资区间校验 `min <= max`
- 后台侧策略：先生成“可查看+可停用”，编辑功能后续按学生端接口收口

## 4.6 `t_enterprise_info`（企业审核）

- 查询条件：`enterprise_name`（LIKE）、`audit_status`（EQ）、`create_time`（BETWEEN）
- 列表展示：`enterprise_name`、`unified_code`、`contact_name`、`contact_phone`、`audit_status`、`audit_time`
- 表单建议：
  - `license_url` 必填，文件上传控件
  - `intro` 多行文本
- 审核字段控制：
  - `audit_status/audit_reason/audit_by/audit_time` 仅管理员审核接口可改
  - 企业用户不允许直改审核状态

## 5. 页面操作权限与按钮可见规则

## 5.1 企业端（`enterprise:*`）

- 岗位页面：
  - `enterprise:post:add`：显示“新增岗位”
  - `enterprise:post:edit`：仅 `post_status in (0,2)` 显示“编辑”
  - `enterprise:post:up`：仅 `post_status in (0,2)` 且企业审核通过显示“上架”
  - `enterprise:post:down`：仅 `post_status='1'` 显示“下架”
- 投递页面：
  - `enterprise:application:hire`：仅 `application_status='0'` 显示“录用”
  - `enterprise:application:complete`：仅 `application_status='1'` 显示“标记完成”
- 互评页面：
  - `enterprise:evaluation:add`：仅 `application_status='2'` 且不存在企业评学生记录时显示

## 5.2 学生端（`student:*`）

- 岗位列表：
  - `student:collection:add`：未收藏时显示“收藏”
  - `student:collection:remove`：已收藏时显示“取消收藏”
  - `student:application:add`：未投递且岗位上架且未过期时显示“投递”
- 投递记录：
  - 当 `application_status='2'` 且未评价企业时，显示“去评价”按钮（`student:evaluation:add`）

## 5.3 管理端（`campus:*`）

- 企业审核：`campus:enterprise:audit:pass`、`campus:enterprise:audit:reject`
- 岗位监管：`campus:job:review:forceDown`
- 统计：`campus:stat:query`、`campus:stat:export`

## 6. 生成后必须立即改造清单（按层）

## 6.1 Controller

1. 企业/学生接口全部按当前登录用户ID做数据隔离（禁止前端传入越权 `student_id/enterprise_id`）。
2. 禁用通用“全字段更新”入口，改为动作型接口：`hire`、`complete`、`evaluate`。
3. 统一错误码与文案：重复投递、越权操作、非法状态流转、提前互评。

## 6.2 Service

1. 落实状态机校验：
   - 投递：`0 -> 1 -> 2`，禁止跳级/回退；
   - 互评仅允许 `application_status='2'`。
2. 落实幂等：
   - 投递接口幂等处理（唯一键冲突转业务错误）；
   - 录用/完成接口要求“条件更新命中1条”才算成功。
3. 补充跨表一致性：
   - 互评入库前校验 `application_id/post_id/student_id/enterprise_id` 一致。

## 6.3 Mapper

1. 新增聚合SQL：岗位投递量、录用量、完成人数统计。
2. 列表查询默认附加 `del_flag='0'`。
3. 对关键更新使用条件更新（示例：`where application_id=? and application_status='1'`）。

## 6.4 前端

1. 通过字典统一状态色与文案，不硬编码中文状态。
2. 操作按钮按“权限+状态”双条件渲染。
3. 动作接口成功后刷新列表，避免页面状态与后端不一致。
4. 对“提前互评”错误码给出明确提示，避免误判为系统异常。

## 7. 接口级状态机与幂等建议

## 7.1 投递状态机

- `已投递(0)` -> `已录用(1)`：仅岗位所属企业可执行；
- `已录用(1)` -> `企业已完成(2)`：仅岗位所属企业可执行；
- 非法流转：直接拒绝并记录审计日志。

## 7.2 互评闸门

- 允许互评前置条件：
  - 投递状态为 `2`；
  - 当前用户属于该投递的学生或企业；
  - 对应评价类型尚未存在。

## 7.3 幂等策略（最小实现）

- `student:application:add`：依赖唯一键 `(post_id, student_id)`；重复请求返回“请勿重复投递”。
- `enterprise:application:hire`：仅当当前状态为 `0` 更新为 `1`。
- `enterprise:application:complete`：仅当当前状态为 `1` 更新为 `2`。
- `*:evaluation:add`：依赖唯一键 `(application_id, evaluation_type)` 防重。

## 9. 常见返工点与规避策略

1. **返工点：生成后仍保留通用编辑接口**
   - 规避：在第1次落地时就拆成动作型接口（录用/完成/互评）。
2. **返工点：按钮只做权限判断，未做状态判断**
   - 规避：统一实现“权限 + 状态”双重可见规则。
3. **返工点：前端传 `student_id/enterprise_id` 造成越权漏洞**
   - 规避：身份字段后端注入，前端不传。
4. **返工点：统计字段允许手工编辑导致数据污染**
   - 规避：统计字段从表单彻底移除，仅服务端聚合更新。
5. **返工点：字典未初始化，页面状态显示空白**
   - 规避：将字典检查加入部署前置项和冒烟清单。
6. **返工点：重复投递/重复互评返回SQL异常**
   - 规避：捕获唯一键冲突并转换为业务错误码。
