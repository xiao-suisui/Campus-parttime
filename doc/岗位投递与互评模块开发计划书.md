# 岗位投递与互评模块开发计划书（4周）

## 1. 计划目标

本计划聚焦“最小可用闭环”业务链路：

1. 企业发布岗位；
2. 学生收藏/投递岗位；
3. 企业处理投递并录用；
4. 企业标记兼职完成；
5. 学生与企业双向互评。

约束说明：

- 状态集采用最小闭环；
- 互评仅在企业将投递记录标记为“企业已完成”后开放；
- 公告功能复用 `sys_notice`、`sys_notice_read`，不新建公告业务表。


## 2. 模块范围

### 2.1 本期包含

- 主数据：学生信息、学生简历、企业信息；
- 业务数据：岗位、投递、收藏、互评；
- 状态字典：岗位状态、投递状态、互评类型、企业审核状态；
- 企业端：岗位管理、投递处理、完成标记、互评录入；
- 学生端：岗位收藏、岗位投递、查看录用结果、互评录入；
- 管理端：企业资质审核（关联企业信息表）。

### 2.2 本期不包含

- 岗位举报、黑名单、岗位置顶、岗位刷新；
- 智能推荐算法深度优化；
- 三方短信、支付能力。


## 3. 关键状态与业务规则

### 3.1 岗位状态（`job_post_status`）

- `0` 草稿
- `1` 上架
- `2` 下架

### 3.2 投递状态（`job_application_status`）

- `0` 已投递
- `1` 已录用
- `2` 企业已完成

### 3.3 互评类型（`job_evaluation_type`）

- `1` 学生评企业
- `2` 企业评学生

### 3.4 核心规则

- 同一学生对同一岗位仅允许一条投递记录；
- 同一投递记录同一评价类型仅允许一条互评记录；
- 当 `t_job_application.application_status = 2` 时，才允许创建互评；
- 投递、收藏、互评均保留软删除字段与审计字段，保障可追溯。

## 5. 4周开发排期

## 第1周：建模与接口骨架

目标：冻结状态模型与数据模型，完成数据库与后端骨架。

- 后端任务
  - 执行并校验 `campus_parttime_module.sql`；
  - 建立 Domain/Mapper/Service/Controller 基础结构；
  - 输出接口清单（入参、出参、分页、错误码）。
- 前端任务
  - 企业端与学生端页面原型图；
  - 明确字段映射、字典映射、状态文案。
- 联调任务
  - 统一时间格式、主键命名、分页模型；
  - 统一异常返回文案。
- 测试任务
  - 编写最小闭环测试用例（正常流与越权流）。

交付物：数据库脚本v1、接口清单v1、状态机说明v1。

验收标准：

- 建表成功，索引与唯一约束生效；
- 互评闸门规则在设计文档中明确。


## 第2周：后端核心功能完成

目标：后端接口可独立跑通全链路。

- 后端任务
  - 岗位：发布、编辑、上下架、列表查询；
  - 投递：创建投递、查询投递、企业录用；
  - 完成：企业标记兼职完成；
  - 互评：创建/查询互评，并校验“仅企业已完成后可评”；
  - 统计：岗位投递量、录用量、完成量聚合。
- 前端任务
  - 企业端页面初版接入：岗位管理、投递处理、完成标记、互评录入。
- 联调任务
  - 使用测试账号完成接口冒烟测试；
  - 校验字典驱动显示与状态回显一致性。
- 测试任务
  - 覆盖重复投递、提前互评、非法状态跳转等异常场景。

交付物：后端API v1、接口测试记录、错误码说明。

验收标准：

- 可通过 API 完成“发布→投递→录用→完成→互评”；
- 提前互评请求必须被拒绝。


## 第3周：页面联调与权限收口

目标：前后端页面可完整演示，权限边界正确。

- 后端任务
  - 补充筛选条件、详情聚合接口；
  - 完成角色权限点与菜单权限绑定。
- 前端任务
  - 企业端完善交互细节；
  - 学生端完成投递记录与互评页面；
  - 公告模块复用 `sys_notice` 展示。
- 联调任务
  - 逐页面联调修复字段错配、状态错配、分页异常；
  - 验证企业/学生/管理员三角色菜单隔离。
- 测试任务
  - 执行联调回归测试与权限回归测试。

交付物：页面功能v1、联调问题清单、权限矩阵v1。

验收标准：

- 页面可端到端跑通全流程；
- 未完成状态下无互评入口。


## 第4周：稳定性与验收

目标：完成缺陷清理、测试报告与答辩演示材料。

- 后端任务
  - 修复 P0/P1 缺陷；
  - 优化慢查询，补全必要索引；
  - 完善日志与异常提示。
- 前端任务
  - 优化空态/加载态/失败态交互；
  - 优化关键文案与操作引导。
- 联调任务
  - 全链路回归，冻结版本；
  - 准备演示数据与演示账号。
- 测试任务
  - 输出测试报告（通过率、遗留问题、风险说明）。

交付物：可演示版本、测试报告、部署说明、演示脚本。

验收标准：

- 三项硬约束全部满足：最小闭环、互评闸门、公告复用；
- 关键流程一次演示通过。


## 6. 风险与应对

1. 状态变更频繁导致返工
   - 应对：第1周末冻结状态模型，后续变更走评审。
2. 并发更新导致状态错乱
   - 应对：关键流转接口增加条件更新与幂等校验。
3. 前后端字段不一致
   - 应对：统一 DTO 命名规范，接口文档与代码同步更新。
4. 进度压缩导致测试不足
   - 应对：第2周开始每日冒烟，第3/4周执行回归清单。
